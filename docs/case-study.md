## 1. 故事的起点

### 1.1 最初的需求

项目的起点非常明确：**做一个 AI 智能筛选功能**。

用户在多维表格的筛选栏里输入一句自然语言（比如"找出本周到期且未完成的高优先级任务"），AI 自动将其转化为结构化的筛选条件，表格即时过滤。

| 传统方式 | AI 方式 |
|---------|---------|
| 手动选字段 → 选操作符 → 填值 → 组合逻辑 | 输入一句话 → 自动生成 |
| 需要理解 15+ 种操作符语法 | 自然语言，零学习成本 |
| 多条件筛选需要 5-10 次点击 | 一句话完成 |

### 1.2 问题来了：筛选功能需要一个表格

AI 筛选不能凭空运行——它需要一张数据表：有字段定义、有数据记录、有视图来承载筛选结果。所以第一步不是写 AI，而是**先搭一个能跑的多维表格底座**。

这个"顺便补齐底座"的模式，后来成了整个项目的核心节奏：**每做一个新功能，就把它依赖的底座能力一起补上**。

### 1.3 开发方式

**人机协作，对话驱动**：产品经理（我）提出需求、审查设计、验收效果；**Claude Code**（AI 编程助手）负责 PRD 编写、架构设计、全栈编码、部署上线。

没有 Jira，没有排期会，没有代码评审流程。一句话描述需求，Claude Code 直接从方案设计到代码实现到部署上线，全程在对话中完成。

## 3. 第一个功能：AI 智能筛选

### 3.1 这是项目真正的起点

Day 1 的核心目标不是"搭一个表格"，而是**让 AI 筛选跑通**。表格只是让筛选有地方可以作用的载体。

所以开发顺序是：

```
1. 搭项目骨架（Express + React）
2. 写 Mock 数据（一张"项目管理"表，10 个字段，20 条记录）
3. 渲染基础表格（能看到数据就行）
4. ★ 对接火山引擎 ARK API，实现 AI 筛选核心链路
5. 顺手补齐：列宽拖拽、列排序、筛选条件回显
```

### 3.2 AI 筛选的技术方案

**调用链路**：

```
用户输入 "找出本周到期的高优先级任务"
  ↓
前端 → POST /api/ai/filter/generate (SSE)
  ↓
System Prompt + User Query → 火山引擎 ARK API
  ↓
AI 调用 get_table_structure → 获取字段列表（名称、类型、选项值）
  ↓
AI 调用 get_sample_data → 获取 3 条示例记录（理解数据格式）
  ↓
AI 输出筛选 JSON:
{
  "logic": "and",
  "conditions": [
    ["截止日期", ">=", "2026-04-14"],
    ["截止日期", "<=", "2026-04-20"],
    ["优先级", "==", "高"]
  ]
}
  ↓
SSE 流式推送 → 前端解析 → 筛选条件回显 + 表格过滤
```

**关键设计决策**：

- **SSE 流式输出**而非 REST 等待——用户能看到 AI 在"思考"，体验更好
- **多轮 Tool Use**而非把表结构塞进 Prompt——表结构可能很大，Tool Use 让 AI 按需获取
- **温度 0.1**——筛选是精确任务，错一个字段名整个筛选就废了
- **输出只有 JSON**——Prompt 里明确"不包含任何其他内容"，避免模型"解释"

### 3.3 为 AI 筛选补齐的底座

为了让 AI 筛选完整可用，顺带补齐了：

| 底座能力 | 为什么需要 |
|---------|-----------|
| 表格渲染 | 筛选结果需要展示 |
| 字段类型系统 | 不同类型有不同筛选操作符 |
| Mock 数据 | AI 需要真实数据来理解表结构 |
| 筛选条件 UI | 用户需要看到/编辑 AI 生成的条件 |
| 列宽/列排序 | 筛选后用户需要正常使用表格 |

## 6. 功能生长：从筛选到更多 AI 能力

上线后我们发现，既然底座已经逐步补齐，可以继续叠加更多 AI 能力。于是项目从"一个 AI 筛选功能"逐步长成了"一个有多个 AI 能力的多维表格产品"。

### 6.1 第二个 AI 能力：字段搜索 + 拼音模糊匹配

**触发点**：字段配置面板做好后，发现字段多了找不到。

**做了什么**：
- 搜索框 + 拼音模糊匹配（`pinyin-pro` 库）
- 输入"rq"能匹配到"日期"字段
- 搜索同时过滤 AI 筛选下拉中的字段列表

**顺带补齐的底座**：字段配置面板的排序、可见性控制。

### 6.2 第三个能力：删除记录二次提示 + Undo

**触发点**：用户误删数据没有挽救手段。

**做了什么**：
- 安全删除保护（可配置的二次确认弹窗）
- 多步撤销栈（Ctrl+Z 连续回退多步操作）
- 前端乐观更新 + 后端确认，失败自动回退 + Toast 提示

**顺带补齐的底座**：
- 行选择（单选、Shift 多选、全选）
- 单元格框选 + Delete 清空
- 双击进入编辑态
- Toast 通知系统
- 右键上下文菜单

### 6.3 第四个 AI 能力：AI 推荐字段

**触发点**：用户新增字段时不知道该加什么。

**做了什么**：
- 新增字段弹窗顶部展示 AI 推荐的字段 chip
- 点击即可一键创建
- 每次展示 6 个（2 行），支持刷新换一批

**技术亮点**：
- **服务端缓存 + 预热**：打开表时后台静默调用 LLM，用户点击"新增字段"时瞬间展示，体感零延迟
- 5 分钟 TTL，每表隔离
- 动态推荐数量（根据已有字段数 10-20 个）
- 排除已展示的推荐，避免重复

**顺带补齐的底座**：
- 新增字段的完整 UI（类型选择器、config 配置面板）
- 18 种字段类型的图标体系（SVG）

### 6.4 第五个 AI 能力：AI 创建数据表

**触发点**：既然 AI 能推荐单个字段，能不能直接生成整张表的结构？

**做了什么**：
- 点击 Sidebar "通过 AI 创建" → Popover 输入表名
- AI 根据表名生成 8-20 个字段（含类型、config、选项颜色等完整定义）
- 三状态 UI：输入态 → 生成中（进度动画）→ 完成态（采用 / 重置为空白）
- 输入"项目管理"→ AI 自动生成：项目名称(Text)、负责人(User)、状态(SingleSelect, 4个选项带颜色)、优先级(SingleSelect)、截止日期(DateTime)、进度(Progress)、预算(Currency)...

**顺带补齐的底座**：
- Table 创建 API（带默认字段 + 空记录 + Grid 视图）
- Table 重置 API（替换所有字段、重建记录和视图）
- 枚举所有字段类型的 config schema（选项颜色值、数字格式、货币精度、评分范围等）
- Document 级 SSE（新建表实时同步到其他用户的 Sidebar）

## 7. Prompt Engineering 实践

三套 Prompt 服务三个 AI 能力，设计策略完全不同。

### 7.1 AI 筛选 Prompt —— 精确性优先

筛选是结构化输出场景，错一个字段名或操作符整个筛选就失败。

**核心策略**：
1. **角色锁定**：你是"多维表格的智能筛选助手"，不是通用聊天
2. **输出约束置顶**（最高优先级）：输出有且只有 JSON，无解释、无 Markdown
3. **操作符枚举**：列出所有合法操作符 + 对应自然语言映射（"不低于" → `>=`）
4. **类型特化规则**：
   - SingleSelect → 值必须是已有选项
   - DateTime → 用当前日期锚定"今天/本周/上个月"
   - User → 模糊匹配用户姓名
5. **空条件兜底**：闲聊/无关请求返回空 JSON 而非拒绝
6. **多轮工具策略**：先 `get_table_structure` 获取结构再生成，不猜

```
温度: 0.1          ← 最低创造性，最高确定性
max_tokens: 4096   ← 防截断
thinking: disabled  ← 直接输出，不浪费 token
```

### 7.2 AI 字段推荐 Prompt —— 创造性与约束并存

推荐要"有创意但合规"——字段名有业务含义，类型在支持列表内。

**核心策略**：
1. 列出所有 28 种支持的字段类型（含 AI 类型如 ai_summary、ai_classify）
2. 已有字段名不能重复推荐
3. 根据表名和已有字段的业务语义推荐互补字段
4. 输出 JSON 数组，每项 `{ name, type }`

```
温度: 0.7   ← 中等创造性
```

### 7.3 AI 建表 Prompt —— 最复杂的结构化输出

一次生成完整表结构，包含字段名、类型、config 配置。任何细节错误都导致创建失败。

**核心策略**：
1. **主键约束**：第一个字段必须是 Text + isPrimary=true
2. **逐类型 config 规则**：18 种字段类型，每种精确定义 config schema
   - SingleSelect：必须有 `options` 数组，3-6 个选项，附颜色值
   - Currency：必须指定 `currencyCode` + `currencyPrecision`
   - Rating：必须指定 `ratingSymbol` + `ratingMin` + `ratingMax`
3. **语义绑定**：包含"姓名"或以"人"结尾的字段 → 强制用 User 类型
4. **不可用类型清单**：Formula、Lookup 等需要引用关系的类型不生成
5. **完整 few-shot 示例**：给出"项目管理"表的完整 JSON，消除歧义
6. **名称卫生**：禁止 `[` `]` 字符（会破坏内部引用语法）
7. **弹性数量**：8-20 个字段，模型根据业务复杂度自行判断

```
温度: 0.7         ← 创造性设计
thinking: disabled ← 直接输出 JSON，加速响应
```

### 7.4 PE 迭代过程

Prompt 不是一次写好的，AI 筛选的 Prompt 经历了多轮迭代：

| 版本 | 问题 | 改进 |
|------|------|------|
| v1 | AI 有时输出解释文字 | 加"输出约束（最高优先级）"段落 |
| v2 | 单选字段生成了不存在的选项值 | 加"值必须是已有选项"规则 |
| v3 | "上周"的日期算错了 | 加 Tool 返回当前日期，Prompt 明确时间锚定规则 |
| v4 | 复杂查询不准确 | 改为多轮 Tool Use，先看结构再生成 |
| v5 | 温度太高导致偶尔幻觉 | 从 1.0 降到 0.1 |

## 8. 工程实践与踩坑

### 8.1 典型的坑

#### IME 输入法 Enter 键冲突
**问题**：中文输入法在组合输入（拼音）阶段按 Enter 选词，被表格误认为"提交编辑"。
**解法**：监听 `compositionstart` / `compositionend`，组合中忽略 Enter。

#### `setSelectionRange` 在 number 输入框白屏
**问题**：双击数字字段进入编辑态时整个页面崩溃。`<input type="number">` 不支持 `setSelectionRange()`。
**解法**：字段类型为 Number 时跳过光标定位。

#### AI 建表的字段推荐竞态
**问题**：AI 建表 = 创建空表 + switchTable + resetTable。`switchTable` 触发字段推荐预热，此时表只有 1 个默认字段，推荐不准确。
**解法**：AI 建表跳过 `switchTable`，直接设置 state，确保字段已经是 AI 版本才更新 activeTableId。

#### Unicode 字符串解析
**问题**：中文弯引号 `"` `"` 在 i18n 文件中被 esbuild 误认为字符串边界，构建报错。
**解法**：使用 Unicode 转义 `\u201C` `\u201D`。

#### SSE 断线重连
**问题**：网络断开再恢复时可能错过事件，数据不一致。
**解法**：EventSource reconnect 后触发 `full-sync`，重新拉取完整数据。

### 8.2 架构决策

| 决策 | 选择 | 备选 | 理由 |
|------|------|------|------|
| 数据存储 | 内存 Map | PostgreSQL + Prisma | 尝试过 PG 迁移但回退——演示项目优先迭代速度 |
| 实时同步 | SSE | WebSocket | 单向推送够用，浏览器原生支持自动重连 |
| i18n | 自建 Context + Hook | i18next | 2 种语言，自建方案零依赖，130 行搞定 |
| AI 建表 | 先建空表再替换 | 一步创建 | 复用已有 createTable，降低改动范围 |
| 字段推荐 | 服务端缓存 + 预热 | 客户端缓存 | 服务端缓存可跨用户复用，体感零延迟 |

### 9.2 Figma → 代码：设计还原是怎么做到的

这是整个项目中最值得展开的工具链。传统流程中设计还原依赖"人看设计稿 → 人写 CSS"，这里变成了 **Claude Code 直连 Figma 设计文件**。

#### 9.2.1 Figma MCP 打通

项目配置了 **Figma MCP Server**（Model Context Protocol），让 Claude Code 可以直接调用 Figma API：

```
Claude Code 对话 → 调用 Figma MCP 工具 → Figma API → 返回设计数据
```

可用的 MCP 工具：
- `get_design_context` — 获取组件的设计上下文（尺寸、颜色、间距、状态）
- `get_screenshot` — 获取指定节点的截图
- `search_design_system` — 在设计库中搜索组件
- `get_metadata` — 获取文件/节点元信息
- `use_figma` — 通用 Figma 查询

项目注册了两个 Figma 设计库：
- **组件库**（UD-03-基础组件-桌面端）File Key: `7rik2X7IeAxfH0qXFklqjb` — 按钮、输入框、弹窗、Toast 等
- **图标库**（UD-07-图标表情库）File Key: `z27mSnJ9vbBeW6VnkLVAg6` — 所有图标的标准 SVG

#### 9.2.2 图标获取流程

需要新图标时，Claude Code 的工作流：

```
1. search_design_system("filter icon") → 找到图标节点 ID
2. get_design_context(fileKey, nodeId) → 获取 SVG 路径数据 + 尺寸
3. 将 SVG 内联写入 React 组件（使用 currentColor 继承颜色）
```

项目中所有工具栏图标、字段类型图标、菜单图标都是这样从 Figma 获取的——不是手动截图描边，是直接拿到矢量路径。

#### 9.2.3 组件样式还原流程

需要新增/修改 UI 组件时：

```
1. get_design_context(组件库 fileKey, 组件 nodeId)
   → 返回：尺寸、内边距、圆角、颜色、字重、行高、状态变体
2. 对照返回的设计 Token 写 CSS
3. 如果有歧义 → get_screenshot(nodeId) 获取截图辅助确认
```

**实际例子**——CreateTablePopover 的完成态样式还原：

> **我**：（发了一张 Figma 截图）"生成完成的样式参考这个"
>
> **Claude Code**：
> 1. 分析截图：居中大尺寸勾选图标、标题文字、两个纵向排列的全宽按钮
> 2. 尝试通过 Figma MCP 获取精确数值（这次 MCP 网络失败了）
> 3. 基于截图目测 + 设计系统规范（`design-resources.md` 中的色值、间距、字重）还原
> 4. 写好后我反馈"按钮左右和下 padding 16px"→ 一次微调搞定

#### 9.2.4 UX Skill：让设计规范自动生效

项目在 `.claude/skills/ux-frontend-design.md` 中定义了一个 **UX & Frontend Design Skill**，Claude Code 在做任何 UI 变更时自动加载。这个 Skill 包含：

- **色彩体系**：主色 `#1456F0`、文字 `#1F2329` / `#646A73` / `#8F959E`、边框 `#DEE0E3`...
- **组件尺寸**：表头 36px、输入框 32px、工具栏按钮 28px、最小列宽 60px...
- **间距规范**：面板 16px、列表项 4px、按钮 4px 8px / 6px 12px...
- **交互模式**：拖拽 4px 死区、编辑触发规则、选择互斥逻辑...
- **CSS 命名**：`.component-name` + `.prefix-element` + `.element.state`
- **强制要求**：每次 UI 变更前必须从 Figma MCP 获取最新设计规范

这意味着 Claude Code **不是凭记忆写 CSS，而是每次都有一份完整的设计规范作为约束**。这是设计还原度高的根本原因。

#### 9.2.5 当 MCP 失败时的降级方案

Figma MCP 不是 100% 可用的——我们遇到过多次 `net::ERR_FAILED` 网络错误。降级策略：

1. **用户发截图**：产品经理直接截 Figma 设计稿发到对话中
2. **设计资源文档**：`docs/design-resources.md` 记录了完整的设计 Token（色值、间距、圆角、阴影...）
3. **UX Skill 兜底**：即使 MCP 不可用，Skill 中的设计规范仍然生效

这三层保障让设计还原不依赖单一渠道。

### 9.3 对话驱动开发

整个项目的开发方式：

```
产品经理描述需求（自然语言 / 截图 / Figma 链接）
        ↓
Claude Code 分析需求、设计方案
        ↓
Claude Code 编写代码（前端 + 后端 + Prompt + CSS）
        ↓
Claude Code 构建 + 部署到服务器
        ↓
产品经理在线上验收 → 反馈调整 → 循环
```

**典型一轮对话——Bug 修复**：

> **我**："双击某个数字字段的单元格，预期应该是进入编辑态，但会直接白屏，检查下原因"
>
> **Claude Code**：读取源码 → 定位到 `setSelectionRange` 调用 → 发现 `<input type="number">` 不支持该 API → 修复代码 → 构建 → 部署上线

从 bug 报告到修复上线，一个对话回合。

**典型一轮对话——设计还原**：

> **我**：（发一张 Figma 截图）"生成完成的样式参考这个"

> **Claude Code**：分析截图中的布局结构 → 对照 UX Skill 中的设计 Token → 重写 CSS + JSX → 部署

设计还原也在对话中完成，不需要切换工具或单独写 CSS spec。

## 11. 给产品经理的指南：怎样写 PRD 让 AI 一次交付

这个项目中，有些功能一轮对话就交付了，有些功能来回调了五六轮。复盘下来，**返工几乎都是因为需求描述不够具体**。以下是我总结的经验——产品经理写什么样的需求，可以让 Claude Code 一次完成开发、自测和验收，直接交付可上线的代码。

### 11.1 核心原则：像写给一个远程外包一样写

Claude Code 极其能干，但它**没有你脑子里的画面**。你觉得"显而易见"的东西，它可能完全不知道。所以需求描述要像写给一个从未见过你产品的远程外包：**所有你脑子里想象的交互细节，都要写出来。**

### 11.2 好需求 vs 坏需求对比

#### 坏需求："做一个删除确认弹窗"

结果：Claude Code 会给你一个能用但不一定符合预期的弹窗——位置、大小、文案、按钮顺序、动画可能都需要调。

#### 好需求：

```
做一个删除记录的确认弹窗：

触发方式：
- 选中行后按 Delete 键
- 右键菜单选择"删除"

弹窗规格：
- 宽 400px，居中遮罩 rgba(0,0,0,0.4)
- 圆角 12px，阴影参考 design-resources.md

内容：
- 标题："删除 N 条记录"（N 为选中数量）
- 描述："删除后可通过 Ctrl+Z 撤销"
- 按钮：[取消（次按钮）] [删除（危险按钮 #D83931）]
- 按钮靠右排列，间距 8px

行为：
- Enter 键 = 确认删除
- Escape 键 = 取消
- 点击遮罩 = 取消
- 删除后 Toast 提示"已删除 N 条记录"

有一个"安全删除"开关（localStorage 持久化），关闭后跳过弹窗直接删除。
```

#### 11.3 需求描述的五个维度

经过这个项目的实践，一个能让 Claude Code 一次交付的需求应该包含：

**1. 触发条件**——什么操作触发这个功能
```
✗ "用户可以创建表"
✓ "用户点击 Sidebar 底部的 + 按钮，弹出菜单，菜单第一项'数据表'可点击创建"
```

**2. 视觉规格**——尺寸、颜色、间距、布局
```
✗ "弹窗里有个输入框和按钮"
✓ "Popover 313px 宽，内边距 16px。标题 14px #1F2329 字重 500，
   标题下方 12px 是输入框（32px 高，placeholder '请输入表名称'），
   输入框下方自动填满间距，底部 16px padding 处是按钮（主按钮样式，满宽）"
```

**3. 状态流转**——从触发到结束的每个状态
```
✗ "AI 生成表结构"
✓ "三个状态：
   1) 输入态：标题 + 输入框 + 创建按钮（输入为空时 disabled）
   2) 生成中：四芒星动画 + 进度条 + '正在生成...' 文案
   3) 完成态：大号勾选图标(48px) + '已创建「表名」' + 两个按钮纵向排列"
```

**4. 边界情况**——异常/极端场景怎么处理
```
"生成失败 → 显示错误提示，可重试"
"生成中点击外部 → abort 请求，关闭 Popover"
"输入超长表名 → 输入框内滚动，不换行"
"网络断开 → Toast 提示，不崩溃"
```

**5. 与已有功能的关系**——新功能如何与现有系统交互
```
"创建完成后：
 - 新表出现在 Sidebar 列表末尾
 - 自动切换到新表
 - 其他用户的 Sidebar 通过 SSE 实时更新
 - AI 字段推荐需要对新表触发预热"
```

### 11.4 产品经理应当具备的工程师思维

写好需求只是一半。这个项目让我意识到，和 AI 编程助手协作时，产品经理需要**具备一些工程师思维**——不是要你写代码，而是要理解"代码世界的运行规则"，这样你才能真正驾驭 AI 助手，而不是被它带着走。

#### 11.4.1 技术方案优先——先对齐"怎么做"再动手

**教训**：这个项目中返工最严重的一次，是 AI 创建数据表功能。我直接说"做一个 AI 建表"，Claude Code 就开始写代码了。写完后我才发现它的方案是"一步到位创建"，但我想要的是"先建空表再用 AI 字段替换"——因为后者可以复用已有的 createTable 逻辑。结果推倒重来。

**正确做法**：在任何非平凡功能开始编码前，先让 Claude Code **出技术方案**（用 Plan 模式），对齐以下几点再动手：

```
1. 数据流：数据从哪来、经过哪些处理、存到哪里
2. API 设计：新增/修改哪些端点，请求和响应格式
3. 组件结构：新增哪些组件，和已有组件的关系
4. 状态管理：哪些状态需要新增，放在哪个层级
5. 复用策略：哪些已有能力可以复用，避免重复造轮子
```

这不需要你懂代码细节，但你需要能**看懂方案、判断合理性、提出质疑**。比如：
- "这个方案要新建 3 个 API，能不能复用现有的 createTable？"
- "字段推荐和 AI 建表用的是同一个模型吗？参数有什么区别？"
- "这个状态放在 App.tsx 还是 Sidebar 里？切换表的时候会不会丢？"

**产品经理不需要写技术方案，但需要有能力审技术方案。**

#### 11.4.2 记得打日志——AI 调用必须可观测

**教训**：项目初期 AI 筛选偶尔不准确，但我们无法定位是 Prompt 问题还是模型问题——因为没有日志。后来加了 AI 调用日志（`backend/logs/AI 日志.log`），记录每次调用的 Prompt、Tool Use 过程、模型返回、耗时，问题一目了然。

**产品经理需要主动要求**：

```
"这个 AI 功能上线前，确保以下内容有日志：
 - 每次 AI 调用的输入（用户 query + 表结构摘要）
 - 模型返回的原始内容
 - Tool Use 的调用链路
 - 调用耗时
 - 错误和异常
 日志写到 backend/logs/ 目录，带 GMT+8 时间戳"
```

你不需要知道 `fs.appendFileSync` 怎么写，但你需要知道**没有日志的 AI 功能就是黑盒**——出了问题你连去哪看都不知道。

同理，**API 请求日志**也应该在项目早期就加上（本项目 Day 3 加的请求日志中间件），而不是等到出 bug 才补。

#### 11.4.3 记得及时归档——文档是 AI 助手的长期记忆

**教训**：项目前两天没有写任何文档。到 Day 3 开新对话时，Claude Code 完全不知道之前做了什么、架构是什么样的。我不得不花大量时间重新解释上下文。从 Day 3 开始建立文档体系后，这个问题彻底消失。

**产品经理需要在以下时机要求归档**：

| 时机 | 归档什么 | 归到哪里 |
|------|---------|---------|
| 新功能上线后 | 功能设计（PRD + 技术方案 + Edge Cases） | `docs/design.md` |
| 每次部署后 | 部署记录（日期、commit、改动点） | `docs/changelog.md` |
| 架构/目录变更后 | 项目结构、命令、关键约定 | `CLAUDE.md` |
| 新增设计模式后 | 色值、组件规范、交互模式 | `docs/design-resources.md` |
| 新增测试场景后 | P0/P1 测试用例 | `docs/test-plan.md` |

**关键认知：文档不是给人看的，是给 AI 看的。**

Claude Code 每次新对话都会读取 `CLAUDE.md`。如果这个文件是最新的，它就能在 10 秒内恢复完整的项目上下文。如果这个文件是过时的，你就要花 10 分钟重新解释。

本项目在 `CLAUDE.md` 中维护了一个**部署检查清单**，每次发版前强制检查文档是否同步更新：

```markdown
## Deployment Checklist
- [ ] 更新 CLAUDE.md — 如有架构/结构/命令变更
- [ ] 更新设计文档 — docs/design.md 对应功能模块
- [ ] 更新测试用例 — docs/test-plan.md 新增/修改功能
- [ ] 更新设计资源 — docs/design-resources.md 新增颜色/组件
- [ ] 更新更新日志 — docs/changelog.md 添加发布记录
```

这个清单不是形式主义——它保障了下一次对话、下一个开发者（或同一个 AI）能无缝接手。

#### 11.4.4 发现可复用的经验时，立即总结成 Skill

**教训**：项目前期每次做 UI 变更，都要在对话中重复说"用 #1456F0 作为主色"、"间距 16px"、"圆角 4px"……同样的话说了十几遍。后来把这些整理成 `.claude/skills/ux-frontend-design.md`（UX Skill），之后 Claude Code 做 UI 变更时**自动加载这个 Skill**，再也不用重复说了。

**什么时候该创建 Skill**：

当你发现自己**第三次在对话中重复同一类指令**时——就是创建 Skill 的信号。

```
第 1 次说 "主色用 #1456F0"  → 正常
第 2 次说 "主色用 #1456F0"  → 可以接受
第 3 次说 "主色用 #1456F0"  → 停下来，创建 Skill
```

**怎么创建**：直接告诉 Claude Code：

```
"把我们这个项目的前端设计规范整理成一个 Skill 文件，
 放在 .claude/skills/ux-frontend-design.md，
 包含色彩体系、间距规范、组件尺寸、交互模式、CSS 命名规则。
 以后做 UI 变更时自动参考这个文件。"
```

Claude Code 会帮你生成完整的 Skill 文件。之后每次涉及 UI 变更的对话，它会自动加载这个 Skill——你再也不用说"主色是什么"。

**本项目创建了哪些 Skill**：

| Skill | 内容 | 效果 |
|-------|------|------|
| `ux-frontend-design.md` | 色彩、间距、组件尺寸、交互模式、Figma 库 Key、CSS 命名、动画规范 | UI 变更零重复沟通 |
| `api-conventions.md` | URL 命名、HTTP 方法、请求/响应格式、SSE 事件格式、EventBus 模式、ID 生成规则 | 新增 API 自动遵循已有规范 |
| `ai-prompt-patterns.md` | 温度选择策略、结构化输出模板、Tool Use 模式、Prompt 结构、调试指南 | 新 AI 功能的 Prompt 一次写对 |
| `deployment.md` | 服务器信息、部署命令、PM2/Nginx 配置、回滚策略、健康检查、常见问题 | 部署操作零记忆负担 |

**Skill 的本质是"把产品经理的审美和判断固化成规则"。** 你不可能每次都盯着 AI 写的每一行 CSS。但如果你的审美已经被编码成了 Skill，AI 就会自动按你的标准来——这比你逐像素审查高效 100 倍。

#### 11.4.5 每次提交部署，记得让 AI 自己跑一遍 P0 用例

**教训**：项目中期有一次部署后，发现 AI 筛选功能完全不能用——原因是一个看似无关的重构改动破坏了筛选结果解析。如果部署前跑了 P0 用例，这个问题 30 秒就能发现，而不是上线后用户来报。

**核心认知：Claude Code 能写代码，也能测代码——但你不说，它就不测。**

传统开发中，"提测"是一个明确的流程节点。和 AI 协作时，这个节点很容易被跳过——因为改完代码就能直接部署，太方便了，方便到忘了验证。

**正确做法**：在 `docs/test-plan.md` 中维护 P0 用例清单，每次部署前让 Claude Code 自己跑一遍：

```
"改完之后，先跑一遍 test-plan.md 里的 P0 用例再部署"
```

或者更彻底——写进 CLAUDE.md 的部署检查清单（本项目就是这样做的）：

```markdown
## Deployment Checklist
- [ ] P0 用例全部通过 — 跑一遍 docs/test-plan.md 中所有 P0 用例
```

**Claude Code 怎么"跑"P0 用例**：

它不需要 Selenium 或 Cypress。Claude Code 可以通过 Preview 工具直接操作浏览器：
1. 启动 dev server（`preview_start`）
2. 截图查看页面状态（`preview_screenshot`）
3. 点击按钮、填写输入框（`preview_click`、`preview_fill`）
4. 检查控制台有无报错（`preview_console_logs`）
5. 检查网络请求（`preview_network`）
6. 检查页面文本内容（`preview_snapshot`）

这不是完整的 E2E 测试框架，但足以覆盖"功能核心链路能不能跑通"这个 P0 级别的验证。比如：

| P0 用例 | Claude Code 怎么验证 |
|---------|---------------------|
| 页面加载后表格正确渲染 | `preview_snapshot` 检查表头和数据行是否存在 |
| AI 筛选输入后生成条件 | `preview_fill` 输入查询 → `preview_click` 提交 → `preview_snapshot` 检查筛选条件回显 |
| 新建数据表出现在 Sidebar | `preview_click` 点击新建 → `preview_snapshot` 检查 Sidebar 列表 |
| 删除记录弹出确认框 | `preview_click` 选中行 → 模拟 Delete 键 → `preview_snapshot` 检查弹窗 |

**P0 用例的维护时机**：

每做完一个新功能，就往 `test-plan.md` 里加对应的 P0 用例。不需要写得很详细——一行描述 + 预期结果就够了。关键是**功能和测试同步增长**，不要等到上线出问题才想起来"应该测一下"。

**本项目的 P0 用例现状**：

`docs/test-plan.md` 中包含 8 个模块、50+ 条 P0 用例，覆盖表格视图、单元格选择、行删除、撤销、AI 筛选、实时同步、国际化等所有核心功能。每条用例都有明确的操作步骤和预期结果，Claude Code 可以直接按照用例描述执行验证。

### 11.5 用截图/Figma 链接代替文字描述布局

这个项目中效率最高的需求形式是：**一张截图 + 几句补充说明**。

```
（发 Figma 截图）
"按照这个样式，但按钮的左右和下 padding 改成 16px"
```

比纯文字描述省 80% 的沟通成本。Claude Code 是多模态的，它能直接"看懂"截图中的布局、间距、颜色、层级关系。

**最佳实践**：
- Figma 截图标注好状态（正常态、hover 态、active 态）
- 如果有动画，用文字描述（"进入时从上方 fade in，0.3s ease"）
- 对截图中不确定的地方补充文字说明

### 11.6 用 Figma 设计库 + Skill 消除来回对齐

本项目设计还原来回调整最少的功能，都是在 CLAUDE.md 和 Skill 中预先写好了设计规范的。

**推荐做法**：项目启动时就建立好这些文件——

1. **`docs/design-resources.md`** — 完整设计 Token（色值、间距、字重、圆角、阴影、组件尺寸）
2. **`.claude/skills/ux-frontend-design.md`** — 交互模式、CSS 命名、动画规范、Figma 库 Key
3. **`CLAUDE.md`** — 引用上述文件，标注"设计与代码不一致时以 Figma 为准"

这三个文件建好后，Claude Code 做任何 UI 变更都会**自动查阅设计规范**，而不是凭感觉写 CSS。

### 11.7 一句话也能交付——但前提是上下文足够

这个项目后期，很多需求只用一句话：

> "sidebar 也需要增加一个锚点逻辑，当左侧菜单太长时，应该自动定位到当前选中的表"

这句话就够了，因为：
- Claude Code 已经知道 Sidebar 的组件结构（CLAUDE.md + 代码上下文）
- 知道 `activeItemId` 是当前选中的表（之前的对话/代码）
- 知道 `scrollIntoView` 是标准做法（工程知识）

**一句话够用的前提是项目上下文已经积累充分。** 项目初期需要写详细需求，项目后期一句话就行——因为 CLAUDE.md、Skills、Memory 和之前的代码已经构成了完整的上下文。

### 11.8 反模式：导致返工的写法

| 反模式 | 为什么导致返工 | 怎么改 |
|--------|--------------|--------|
| "做一个弹窗" | 缺少尺寸、布局、行为 | 补充五个维度 |
| "参考飞书的样式" | Claude Code 不知道你指的是哪个版本的飞书 | 给截图或精确色值 |
| "改一下那个间距" | "那个"指哪个？ | "CreateTablePopover 的标题和输入框间距改成 12px" |
| "这个不太对" | 不知道哪里不对 | "按钮应该纵向排列而不是横向，参考这张图" |
| "做好了部署一下" | 没说要不要测试 | "改完先部署，不用测试" 或 "改完先不部署" |
| 需求中途变化但不说明 | Claude Code 按旧理解继续做 | "之前说的 XX 改成 YY，因为 ZZ" |

### 11.9 理想的需求模板

```markdown
## 功能名称
（一句话描述）

## 触发方式
- 用户做什么操作触发
- 入口在哪里

## 视觉设计
- （截图/Figma 链接）
- 补充说明：尺寸、间距、颜色的调整

## 状态流转
1. 状态A → 触发条件 → 状态B
2. 状态B → 触发条件 → 状态C

## 边界情况
- 异常场景1 → 处理方式
- 异常场景2 → 处理方式

## 与现有功能的关系
- 对 XX 功能的影响
- 需要同步更新的地方（SSE、i18n、缓存等）

## 验收标准
- 场景1：操作 → 预期结果
- 场景2：操作 → 预期结果
```

**写到这个程度，Claude Code 基本可以一次交付——写代码、处理边界、补 i18n、更新 SSE 同步，直接给你可上线的代码。**
